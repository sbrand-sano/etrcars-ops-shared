## 概要

本章では、車両顧客管理システム「ETRCARS」の運用における負荷軽減および最適化のための方針、実施計画、関係者の役割、具体的な自動化・効率化策、効果測定方法について記載する。

## 7.1 目的および背景

ETRCARS システムの運用に伴い発生する AWS やアプリケーションの監視、保守、障害対応等の日常作業について、自動化および効率化を推進することで、運用コストや人為的ミスの削減を目指す。特に、車検モジュールにおける画像ロード遅延（0.6 秒）を契機として、システム全体の処理負荷分散と監視自動化の強化が必要とされている。

## 7.2 対象範囲と関係者

本章の対象範囲は以下のとおり。

- AWS インフラの監視およびアラーム通知
- ユーザー操作ログの収集と自動抽出
- ECS Fargate・Lambda を活用したバッチ運用
- Trello による障害・変更・リリース管理

関係者および主な役割は以下のとおり。

| 担当 | 主な役割 |
|---|---|
| エスブランド | 運用設計、承認、監査、CODIUM 社の月次報告確認、ENEOS トレーディングへの報告 |
| CODIUM | AWS バッチ／ログ管理、自動復旧構成、障害一次対応、監視設定変更 |

## 7.3 負荷削減基本方針

- **自動化優先**: 再現可能な業務をスケジューラや Lambda、通知連携で自動化。
- **可視化**: Trello やダッシュボードで各工程を見える化。
- **簡素化**: Trello 承認プロセスを標準化し、人為的操作を削減。
- **リスク管理**: 自動処理前にバックアップと検証を実施。

## 7.4 自動化設計および効率化策

以下の項目について自動化を強化する。

| 項目 | 自動化内容 |
|---|---|
| ECS Fargate | タスク失敗時は ALB 健全性をトリガーに自動再起動 |
| RDS | Multi-AZ 構成によりフェイルオーバー時は自動切替 |
| 通知連携 | CloudWatch アラームからメール通知、Trello 自動起票 |
| GitLab | バージョン・タグ情報の自動連携による構成確認効率化 |
| AWS Inspector | ECS・EC2 のパッチ適用状況の自動検出・報告 |
| MDM | 開発 PC・iPad のソフトウェア・端末の自動更新・管理 |

## 7.5 定型業務テンプレート化

- **Trello テンプレート管理**: 修正、バグ、改修、障害対応、バッチ確認、月次報告などすべてをカードテンプレート化し、承認フローを「依頼 → 承認 → 実施 → 報告 → 完了」で統一。
- **Excel 管理**: AWS 監視、ログ確認、バッチ実行結果を 1 ファイルに統合。

## 7.6 現状分析および負荷削減目標

システムは **導入店舗 200 店規模**を想定し設計されており、CloudWatch でリソース負荷を監視している。現時点で性能上の問題はないが、導入件数増加に応じてリソース設定を見直し、月次でコスト最適化を行う。

> **「店舗」と「アカウント」の関係**: 導入ディーラー → 店舗 → アカウント（店舗 1 つにつきアカウント 1 つ）。第5章 §5.7.3 で「170 アカウント」は同時点での店舗数を表す。

## 7.7 実施計画

CODIUM およびエスブランドによる運用軽減に加え、ENEOS トレーディング本社の監視運用効率化も含めて計画する。現行運用の主な課題は以下のとおり。

- 監視およびログ確認に人的工数がかかり、異常検知の初動が遅れる。
- ユーザー操作ログの収集・分析が手動で、データ量増加により処理負荷が上昇。
- パッチ適用や監視報告作業が個別対応で、作業ミスのリスクが存在。

### 7.7.1 システム負荷削減目標（CODIUM／エスブランド実施）

| 項目 | 目標内容 |
|---|---|
| リソース最適化 | AWS リソースを分析し、過剰な CPU／メモリ割当を削減。必要時に Fargate 設定を見直しコスト最適化。 |
| 自動化強化 | CODIUM によるログチェックやパッチ適用を AWS Lambda と Systems Manager で自動化し、CloudWatch に履歴保存。 |

### 7.7.2 監視運用負荷削減目標（ENEOS トレーディング本社向け）

| 項目 | 目標内容 |
|---|---|
| ユーザー操作ログの監視効率化 | 特約店単位の時間別閲覧から Web 管理ダッシュボード上でログを項目別収集・CSV 出力可能にする |
| アラート定義の設計 | 大量ログをもとに異常動作を定義し、しきい値に基づく自動アラート通知を導入 |
| 分析・可視化強化 | 本社が CloudWatch Logs Insights により特約店別利用状況を可視化 |
| 報告プロセス自動化 | 操作ログの週次集計を自動送信し、エスブランドおよび ENEOS で同一データを共有 |

## 7.8 負荷削減実施手順・使用ツール・リスク管理・効果測定

### 7.8.1 負荷削減手順

1. 稼働中のプロセスを分析し、リソース消費の多い項目を特定。
2. モニタリングで動作を観察し、必要に応じて改善調整を実施。
3. ロードバランサーを設定し、トラフィックを複数サーバに分散。
4. 必要時にサーバを追加して負荷集中を防止。
5. トラフィックの均衡を監視し、偏りがある場合は調整。

### 7.8.2 使用ツール

- AWS CloudWatch
- Application Load Balancer
- AWS Lambda

### 7.8.3 注意事項およびリスク管理

- 負荷削減作業前にシステム全体のバックアップを取得。
- 変更が他システムへ与える影響を考慮。
- テスト環境で事前検証を行い、本番環境への影響を最小化。
- 作業中および作業後は CloudWatch でパフォーマンスを継続監視。
- ENEOS トレーディングおよびエスブランドへ作業前に通知し、情報共有を徹底。

### 7.8.4 効果測定方法

- 自動化対象作業の月次作業時間を可視化し、削減率を算出。
- Trello や運用報告書を用いてエラー件数・応答時間を評価。
- 年次監査にて改善効果を定量的に測定し、次年度計画に反映。

## 7.9 効果測定および改善サイクル

- 自動化率・監視精度・対応時間短縮を四半期ごとに測定。
- エスブランドが CODIUM からの報告をレビューし、ENEOS トレーディングへ共有。
- 改善提案は Trello 内「改善カード」として登録し、次期監査に反映。
