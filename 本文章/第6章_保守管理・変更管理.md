## 6.1 目的および適用範囲

本章では、ETRCARS システムの運用・保守における変更管理および保守統制を定義する。AWS 環境、Web アプリケーション、iPad アプリ、外部サービス（Cloudflare、GitLab 等）に関する保守作業、構成変更、アカウント管理、承認フローを一元管理することを目的とする。

## 6.2 保守対象範囲

本章の保守対象範囲は以下のとおり。

- AWS リソース（ECS、RDS、S3、ALB、CloudWatch、GuardDuty）
- Web アプリケーション（Django）
- **iPad アプリ**:
  - ENEOS 提供 iPad: ENEOS サポートが保守。MDM 導入済み。
  - 特約店自前の iPad: CODIUM の保守対象外。
  - （第1章 §1.4.2・第10章と整合）
- Cloudflare 設定および DNS 構成
- GitLab／CI/CD パイプライン設定
- 「AWS設定変更/パッチ適応」ボード

## 6.3 保守作業分類

| 区分 | 内容 | 実施者 | 承認者 |
|---|---|---|---|
| 定期保守 | セキュリティパッチ・暗号化キー変更・監視設定更新 | CODIUM | エスブランド |
| 障害対応保守 | 障害発生時の再設定・パラメータ修正 | CODIUM | エスブランド／ENEOS |
| 改修保守 | 機能改善や軽微なコード修正 | CODIUM | エスブランド／ENEOS |
| 変更管理 | AWS 設定変更・ネットワーク構成変更 | CODIUM | エスブランド（本社承認） |

## 6.4 構成変更管理方針

すべての構成変更は Trello で一元的に管理する。各案件は「変更管理カード」として登録し、案件番号、変更内容、実施者、承認者、リスク評価、実施日、完了報告を記録する。AWS 設定変更は Trello で承認後に適用する。

> 手順については手順書 11「AWS 基盤変更手順」を確認し実施し、別紙 E9「変更管理台帳」に記載する。

## 6.5 Trello 変更管理ワークフロー

| フェーズ | 内容 | 担当 | チェック項目 |
|---|---|---|---|
| 案件登録 | 変更内容をカード登録 | CODIUM | 内容・影響範囲確認 |
| 承認依頼 | エスブランド承認者にアサイン | CODIUM | リスク評価・承認依頼 |
| 承認／却下 | エスブランド／ENEOS が判断 | エスブランド | 承認チェック |
| 実施 | AWS／GitLab 設定変更 | CODIUM | 変更ログ登録 |
| 検証 | 動作確認を実施 | エスブランド | 結果記録 |
| 完了報告 | Trello カードを完了へ移動 | CODIUM | 添付ログ・承認履歴確認 |

## 6.6 アカウント管理および棚卸し

AWS IAM、GitLab、Cloudflare、各システムのアカウントを定期的に棚卸し・見直しする。

> **アカウント棚卸し頻度の正式定義は第3章 §3.7.7 を参照。**（ユーザー ID: 1 年に一度以上／管理者・特権 ID: 6 ヶ月に一度以上＋月次レビュー／システムアカウント: 半年に一度）

強権限・管理者権限については、別途利用履歴を記載する（「AWS 特権 ID 利用記録」「ETRweb 管理者 ID 利用記録」を使用）。

### AWS の特権 ID

- CODIUM 社は月に 1 度、AWS 特権 ID 利用記録と CloudTrail の利用ログを確認する。
- 不正な利用が発見された場合は、即座にエスブランドへ報告する。
- エスブランドは ENEOS トレーディングと協議し、理由が判明しない場合、アカウントの停止またはパスワードの無効化を実施する。

> 詳細は別紙 E3「PJ アカウント（アカウント棚卸し台帳）」を参照。アカウントの変更についても別紙 E3 に記載し管理する。

## 6.7 AWS 設定変更管理

AWS 設定変更は Terraform を用いてコード化し、GitLab で変更履歴を管理する。セキュリティグループ、IAM ロール、ECS 設定等の変更内容はすべてレビュー承認後に本番環境へ反映する。

## 6.8 保守記録・監査

保守・変更作業の記録はすべて Trello および GitLab に保管する。定期監査はエスブランドにより年 1 回実施され、変更記録・承認履歴・アクセス権限・特権 ID が確認される。

- 変更履歴の詳細は別紙 E9「変更管理台帳」および別紙 E3「PJ アカウント」を参照。
- 開発用 PC・検証用端末は 3 年を目安に更新し、機器情報は別紙 E4「ハードウェア管理」で管理する。
- 廃棄時にはデータ消去証跡を保管する。

## 6.9 ドキュメント改訂手続き

ETRCARS 運用ドキュメント（内部統制書・本文章 11 章・手順書・別紙）の改訂は、本章の変更管理ワークフロー（§6.5）を適用する。

| 区分 | 内容 | 承認者 |
|---|---|---|
| 軽微改訂（誤字・参照名修正） | 校正・形式統一・参照リンク更新 | エスブランド |
| 通常改訂（節の追加・記述変更） | 1 つの章・手順書を対象とした内容更新 | エスブランド（ENEOS 共有） |
| 大規模改訂（章の新設・廃止、定義変更） | 障害レベル定義／ライフサイクル等の正式定義の変更 | ENEOS トレーディング事業戦略部（承認） |

### 改訂時の必須作業

- 改訂履歴表に改訂日・版数・改訂内容・作成者を記載
- 別紙 E9「変更管理台帳」に文書改訂カードを登録
- 関連章・手順書・別紙との整合を確認（特に第5章・第8章・第10章の正式定義との整合）
- 必要に応じて、CODIUM へ共有
