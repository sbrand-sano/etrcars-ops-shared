## 第1章 システム全体概要

### 1.1 目的

本書は、車両顧客管理システム「ETRCARS」の運用・保守業務における基本方針、体制、運用手順、監査および記録管理を定義しています。運用業務の標準化、安定稼働、セキュリティ確保、障害復旧、継続的な改善を体系的に実施できる仕組みの確立を目的としています。また、IPA非機能要求グレード2018「運用・保守性」および ISMS（ISO/IEC 27001）の要求事項に準拠し、システム運用の信頼性、保守性、可用性、完全性の確保を重視しています。

### 1.2 適用範囲

本書の対象は ETRCARS システムの運用および保守に関する全活動を含みます。具体的には、iPadアプリと Web 管理画面の運用管理、RDS・S3・ログ管理のバックアップと保全、AWS（ECS, RDS, S3, CloudWatch, CloudTrail, GuardDuty 等）の監視と構成管理、アクセス制御や脆弱性対策、パッチ適用、監査ログの取得と分析、CODIUM 社が運用する開発・検証環境の構成・変更管理、ENEOS トレーディングによる店舗導入・運用支援・問い合わせ対応、エスブランドマネジメントによる運用統括・監査・リスク管理・改善活動を含みます。

### 1.4 対象システムの概要

ETRCARS は、ENEOS トレーディング株式会社および株式会社エスブランドマネジメントが共同で提供する車両顧客管理アプリケーションシステムです。サービスステーション（SS）の店頭業務効率化、顧客管理精度向上、販売促進支援を目的に設計され、iPad アプリケーションと Web 管理画面で構成されています。顧客・車両情報、販売履歴、見積情報は店頭から登録され、AWS クラウド上の RDS に保存されます。Web 管理画面では抽出・分析・CSV 出力が可能で、DM 発送や販促活動に利用されます。開発・保守・運用はタイ法人 CODIUM Co., Ltd. が担い、AWS 環境の構築・監視・ログ管理・セキュリティ対応を実施しています。

#### 1.4.1 運用対象

| 区分 | 主な機能・内容 | 管理主体 |
|---|---|---|
| iPad アプリ（店頭用） | 顧客／車両情報登録、見積作成、販売履歴登録、App Store 配信 | ENEOS 特約店／CODIUM |
| Web 管理画面（管理用） | 登録データ閲覧、CSV 出力、顧客分析、販促対象抽出 | ENEOS 本社／エスブランド |
| AWS インフラ（サーバー） | ECS、EC2、RDS、S3、ALB、CloudWatch、GuardDuty、CloudTrail、Cloudflare を利用し、アプリ／Web の基盤を構成 | エスブランド／CODIUM |
| GitLab（開発環境） | ソースコードのバージョン管理、開発・リリース管理、ログ追跡 | CODIUM |
| 運用体制 | CODIUM：開発・保守・監視・報告／エスブランド：統括・承認・監査／ENEOS：特約店支援・問い合わせ対応 | 3 社共同 |

#### 1.4.2 前提条件および制約事項

- ユーザーが使用する iPad および PC は、特約店が独自に準備・管理を行う。
  - **ENEOS 提供 iPad**: ENEOS サポートが保守。MDM 導入済み。
  - **特約店自前の iPad**: CODIUM の保守対象外。（C-05 反映、詳細は第6章・第10章を参照）
- 通信回線や端末障害は運用範囲外とする。
- 個人情報の取扱責任は特約店側にあり、利用規約にて同意を取得する。
- AWS 環境のルートアカウントはエスブランドが保有し、運用操作は CODIUM の IAM アカウントにより実施する。
- 個人情報（RDS 内データ）は暗号化されており、運営側（ENEOS・エスブランド・CODIUM）は復号閲覧できない設計とする。
- 特約店は Apple 社提供の最新 iOS セキュリティ対応デバイスを使用することを推奨する。
- システム導入時には個人情報保護・情報セキュリティ教育を実施する。
- **データ・ログ保持期間**:
  - 顧客データ本体（RDS 内）: **現状は恒久保存**。**導入店舗 1,000 店規模到達を目処に正式保持期間を検討予定**（Phase E P-001 で決定）。
  - バックアップデータ（S3）: 7 年間保持。
  - 監査ログ: CloudWatch Logs 30 日（アクティブ参照）→ S3 アーカイブ 7 年 → Glacier 最大 15 年（第8章で正式定義）。
- システム構成は「iPad → Cloudflare（WAF）→ ALB → ECS → RDS → Web 管理画面」で構成される。
- 退会時の個人データの取り扱いについては、別途 ETRCARS 利用規約に基づき実施する。

#### 1.4.3 監視・報告・承認ルート

CODIUM 社が監視・障害検知を実施し、エスブランドマネジメントが内容を確認・承認したうえで、ENEOS トレーディング本社へ定期的に報告します。報告経路は次のとおり。

```
CODIUM（監視・報告） → エスブランド（承認・監査） → ENEOS トレーディング本社（共有・対応判断）
```

#### 1.4.4 AWS 構成および冗長化設計

本システムの AWS 本番環境は東京リージョンを主系、大阪リージョンをバックアップ系とし、アプリケーションは ECS Fargate 上で稼働、ALB による負荷分散を実施します。RDS はマルチ AZ 構成で可用性を確保し、S3 に日次バックアップデータを保存します。冗長構成として RDS（Multi-AZ）、ALB（2 ゾーン）、ECS タスク（2 ゾーン分散）を採用し、S3 は東京／大阪リージョン間でレプリケーションを行います。CloudWatch・GuardDuty・CloudTrail による監視と異常検知を実施し、障害発生時は AWS 運用ドキュメントに基づき大阪リージョンのバックアップより復旧します。

- **復旧試験**: 年 1 回実施（実施月の正式統一は第3章・第5章を参照。C-10／C-14 で章間調整中）。
- **RTO（復旧目標時間）**: 現時点のデータ規模で 2 時間（同一リージョン）／DR シナリオでは 3 時間（第5章 §5.7.1 参照）。
- **RPO（復旧目標時点）**: **24 時間**（RDS 自動スナップショット 0:00 取得を前提）。
