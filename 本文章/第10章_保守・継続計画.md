## 10.1 保守・継続計画の目的

本章は、ETRCARS システム（iPad アプリ、Web 管理画面、AWS インフラ）における長期的な安定運用と災害時の業務継続（BCP）および復旧（DR）体制の確立を目的とする。本計画により、以下を保証する。

- システム障害・災害発生時における迅速なサービス復旧
- 運用・保守に関する責任範囲の明確化
- データの可用性・整合性・機密性の維持
- システム更新・改善・監査に基づく継続的な品質向上

ETRCARS では AWS を主基盤とし、東京リージョン／大阪リージョンを活用したクラウド冗長構成で高可用性を確保。さらに、ネットワーク経路と DNS 管理を Cloudflare で冗長化し、障害発生時の迅速な切替を可能としている。

## 10.2 保守体制と責任分担

保守業務はシステム構成要素ごとに役割を明確にし、ENEOS トレーディング（本社・支店）、エスブランドマネジメント、CODIUM 社が分担して実施する。

| 区分 | 主体 | 主な責任 | 主な業務内容 |
|---|---|---|---|
| 運用統括／保守統制 | 株式会社エスブランドマネジメント | 運用全体の統括および保守方針の策定 | AWS 設定変更承認／障害対応統制／監査・報告書作成 |
| 保守実施／技術対応 | タイ法人 CODIUM Co., Ltd. | AWS 構成変更・監視・障害対応・修正 | ECS／RDS／S3／ALB 構成管理・セキュリティ修正 |
| 運用管理／報告承認 | ENEOS トレーディング株式会社 事業戦略部 | システム利用責任・報告承認・方針決定 | 監査報告確認・再発防止承認・更新計画承認 |
| ユーザー運用支援 | ENEOS トレーディング全国支店 | 店舗支援・一次対応 | アプリ・Web 利用支援／軽微障害の報告・連絡 |

すべての保守作業は Trello「AWS」「改修」「新規開発」ボードで記録・承認され、手順書や証跡（ログ・スクリーンショット）は Trello カードに添付、エスブランドの承認後に完了扱いとする。

### 10.2.1 保守体制と責任分担（要員スキル基準）

ETRCARS システムの保守・障害対応・AWS 監視業務を実施する要員は、以下の技術基準を満たすことを要件として定める。

**(1) AWS インフラ運用スキル**

- AWS の主要サービス（ECS／EC2／RDS／S3／CloudWatch／VPC／IAM／GuardDuty 等）の運用経験を 2 年以上有していること。
- 障害時におけるログ解析（CloudWatch Logs／RDS Performance Insights 等）が可能であること。
- AWS Config／CloudTrail を用いた変更追跡が実施できること。

**(2) インシデント・障害対応スキル**

- 一次切り分け（ネットワーク／アプリ／DB／AWS サービス障害）の判断ができること。
- 再現試験・スモークテスト・復旧確認を実施できること。
- AWS へエスカレーションする判断基準を理解していること。

**(3) セキュリティスキル**

- IAM ロール設計、最小権限原則、KMS 暗号化の原則を理解していること。
- パッチ適用、脆弱性対応（CVSS 評価）、ログ監査の知識を有すること。
- Cloudflare／WAF／セキュリティログの監視ができること。

**(4) DR／BCP 対応スキル**

- クロスリージョンレプリケーション、スナップショット復旧、フェイルオーバーの知識を有すること。
- Terraform／GitLab Pipeline による再構築手順を理解していること。
- DR 切替後の動作確認（API 応答、ECS 正常性、DNS 更新確認）が行えること。

## 10.3 保守対象範囲

ETRCARS システムの保守対象範囲は AWS インフラからエンドユーザー端末、セキュリティ、アプリケーション運用までを含む。

| 分類 | 対象範囲 | 管理責任者 | 備考 |
|---|---|---|---|
| AWS インフラ | ECS、RDS、S3、ALB、CloudWatch、GuardDuty、Config、CloudTrail | CODIUM（実施）／エスブランド（監査） | 東京・大阪リージョンの運用を含む |
| アプリケーション | iPad アプリ、Web 管理画面 | CODIUM（保守）、エスブランド（統括） | バージョン更新・不具合修正含む |
| ネットワーク／DNS | Cloudflare（WAF／DNS／SSL 管理） | CODIUM（実施）、エスブランド（監査） | DDoS 防御・アクセス制御含む |
| 監視／アラート | CloudWatch、Sentry、UptimeRobot、Trello 通知 | CODIUM（運用）、エスブランド（確認） | 死活監視・リソース監視を含む |
| アカウント管理 | AWS IAM／GitLab／Trello アカウント | エスブランド（統括）、CODIUM（実施） | IAM 最小権限原則に基づく管理 |
| データ保護／バックアップ | RDS 自動スナップショット、S3 バックアップ、Glacier 保管 | CODIUM（実施）、エスブランド（確認） | 第8章ライフサイクルに従う |
| MDM／端末管理 | CODIUM 社内 PC、iPad、MDM 登録端末 | CODIUM（実施）、エスブランド（監査） | 紛失・廃棄時のデータ消去含む |

### iPad 保守対象範囲（第1章・第6章と整合）

- **ENEOS 提供 iPad**: ENEOS サポートが保守。MDM 導入済み。
- **特約店自前の iPad**: CODIUM の保守対象外。

## 10.4 ライフサイクル管理

ETRCARS システムでは、クラウドインフラ・アプリケーション・データ資産のライフサイクルを管理し、運用・更新・廃止の各段階で安全性・効率性を確保する。検討は 1 ヶ月前まで実施する。

| 区分 | 内容 | 対応方針 |
|---|---|---|
| インフラ資産 | AWS インスタンス、ストレージ、ネットワーク設定 | AWS Config で変更履歴を管理し、年 1 回棚卸しを実施。利用していないリソースは削除。 |
| ソフトウェア資産 | アプリ／API ソースコード、ライブラリ | GitLab でバージョン管理。サポート切れの計画が判明した場合 1 ヶ月前に、切り替え計画の検討を開始する。 |
| セキュリティ設定 | IAM、WAF、GuardDuty、SecurityHub | 月次点検および脆弱性情報に基づき更新。 |
| バックアップデータ | RDS スナップショット、S3 バックアップ | **第8章ライフサイクル**に従う（S3 7 年 → Glacier 最大 15 年）。 |
| MDM 管理端末 | iPad／PC の資産管理情報 | 半期ごとに棚卸し。退職・破棄時にはデータ完全消去を実施。 |

ライフサイクル管理により、不要リソースの削減・セキュリティリスク低減・コスト最適化を図る。定期的な見直しはエスブランドが実施し、年次監査で ENEOS トレーディングに報告する。

### バックアップデータの削除について

- バックアップデータを論理的に消去する場合、データ消去ツールを使用する等して実施する。
- バックアップデータを物理的に破壊する場合、データの復元が不可能になるよう破壊する。
- 外部委託する際は、委託先からデータ消去を確実に実施した証明を受領する。
- AWS の機能を使用できるものについては、設定にしたがうこと。

## 10.5 継続運用計画（BCP）

災害や AWS 障害時でもサービスを継続するため、以下の BCP（Business Continuity Plan）を定義する。

### 10.5.1 継続運用の目的

- 災害発生時でも主要業務（顧客データ参照・登録）を **72 時間以内**に再稼働させる。
- AWS 東京リージョン障害時は大阪リージョンで代替運用を行う。
- データ損失は **RPO 24 時間**を上限とする（RDS 自動スナップショット 0:00 取得・第5章 §5.7.1 および第1章 §1.4.4 と整合）。

### 10.5.2 継続運用の構成

| 対象 | 継続運用方法 | 備考 |
|---|---|---|
| AWS 東京リージョン障害 | 大阪リージョンの RDS・S3 へフェイルオーバー | RDS クロスリージョンレプリケーションを有効化 |
| DNS 障害／通信断 | Cloudflare を通じて DNS 経路を切替 | ALB エンドポイント変更により DNS 再ルーティング |
| アプリ障害 | ECS 再デプロイによるコンテナ再生成 | Sentry 検知から Trello 報告・再起動実施 |
| Web 障害 | ECS／ALB レベルで対応 | AWS WAF による一時遮断対応。**CloudFront は利用しない**（SSL は Cloudflare で管理） |
| バックアップ喪失 | S3 大阪リージョンからデータ復旧 | Glacier 保管データも復元対象に含む |

### 10.5.3 継続運用の確認

- 四半期ごとに BCP テストを実施し、切替手順と復旧時間を記録。
- 障害時対応の結果は「インシデント報告」ボードに登録し、エスブランドおよび ENEOS 本社が承認。

## 10.6 DR（Disaster Recovery）運用

ETRCARS の DR（災害復旧）構成は、AWS の冗長化およびバックアップ体制に基づく。

### 10.6.1 DR 方針

- 東京リージョン障害時は大阪リージョンのバックアップを使用して再構築。
- 自動レプリケーション済みの RDS データを用い、Terraform スクリプトで環境を再構築。
- DNS 切替は Cloudflare で実施。Cloudflare SSL を切替確認後にシステムを再稼働。

### 10.6.2 DR 対象

| 項目 | 対象 | 復旧手段 |
|---|---|---|
| RDS（データベース） | 本番データ | クロスリージョンレプリケーション＋スナップショット復元 |
| S3（ファイル保管） | 顧客画像・帳票 | バケットレプリケーション＋大阪リージョン復旧 |
| ECS／ALB | Web／API サービス | Terraform 再デプロイ＋ALB 新規エンドポイント発行 |
| DNS／SSL | Cloudflare | CNAME 切替＋Cloudflare SSL 証明書の再発行・確認（**ACM は使用しない**） |

### 10.6.3 DR 運用手順概要

1. 東京リージョンの障害を確認（CloudWatch／GuardDuty）。
2. 大阪リージョンの RDS スナップショットを選択し復旧開始。
3. Terraform で ECS／ALB／S3 構成を自動再構築。
4. Cloudflare DNS を大阪側に切替（SSL 再設定）。
5. システム接続確認（GitLab Pipeline 実行を含む）。
6. Trello に復旧報告を登録し、エスブランド承認後に復旧完了。
