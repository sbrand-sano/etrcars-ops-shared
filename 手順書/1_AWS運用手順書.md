## 監視手順


### 目的

可用性、パフォーマンス、セキュリティ、コンプライアンスを確保するために、AWS RDS、ECS、ALB リソースを継続的に監視する手順を定義します。

### 範囲

次のサーバーでホストされるすべての運用、ステージング、開発ワークロードに適用されます。
AWS RDS – マネージドリレーショナルデータベース
AWS ECS – コンテナ化されたワークロード
AWS ALB（アプリケーションロードバランサー） – Webトラフィック分散

### 手順


#### AWS RDS


##### AWS RDS モニタリング

AWS CloudWatch の拡張モニタリングとパフォーマンスインサイトを有効にします。
監視すべき主要な指標:
CPU使用率、メモリ、ディスクI/O
データベース接続とクエリのレイテンシ
空きストレージ容量とIOPS使用率
重大なしきい値 (1 分以内に 1 つのデータポイントの CPU が 50% を超える、1 分以内に解放可能なメモリが 1 GB 未満) に対してCloudWatch アラームを設定します。
エラー ログ、スロー クエリ ログを収集して確認します。

##### AWS RDS メンテナンス手順

IAM ユーザーでAWS マネジメントコンソールにログインします。
Aurora および RDS → データベース → etrcarsに移動します。
RDSのステータスを確認する
概要タブの下 → ステータス（ 「利用可能」と表示されていることを確認してください）
ログとイベントを確認する
ログとイベントタブ→Cloudwatchアラーム（CPUとRAMの両方が正常に戻っていることを確認してください）
最近のイベントを確認する
ログとイベントタブ→最近のイベント（疑わしいイベントがないことを確認してください）
データベースの負荷を確認する
概要タブ→現在のアクティビティ（赤い線を超えていないことを確認してください）
データベースストレージを確認する
監視タブ→ FreeStorageSpace （ストレージ容量が残っていることを確認してください）
VPC セキュリティ グループの接続を確認する
[接続とセキュリティ]タブ→ [セキュリティ] → [VPC セキュリティ グループ] → [etrcars-rds-sg] を選択 → [受信ルール] (ポートが適切に設定されていることを確認してください)
EC2 サービス→セキュリティ グループ→ etrcars-ecs-sg を選択 → 送信ルール (ポートが RDS に接続できるように適切に構成されていることを確認してください)

#### AWS ECS


##### AWS ECS モニタリング

ECS クラスターとタスクに対してCloudWatch Container Insights を有効にします。
監視すべき主要な指標:
タスクのCPUとメモリの使用率
実行中のタスク数と希望するタスク数
タスクの再起動回数（失敗ループを示す）
サービス展開の健全性（実行中のタスク = 必要なタスク）
CloudWatch アラームを重大なしきい値 (3 分以内に 3 つのデータポイントで CPU > 50%、3 分以内に 3 つのデータポイントでメモリ使用率 > 75%) に設定します。
ECS ログをCloudwatch の集中ログ ソリューションと統合する
ECS タスク定義とイメージの脆弱性のレビューを実行します。

##### AWS ECS メンテナンス手順

IAM ユーザーで AWS マネジメントコンソールにログインします。
Elastic Container Service → etrcars-ecs-cluster → etrcars に移動します。
ECS サービスのステータスを確認する
サービスタブ→ステータス（アクティブになっていることを確認してください）
ECSタスク数を確認する
サービスタブ → etrcars-django-service → 構成とネットワークタブ → Auto Scaling → Desire Count (0より大きいことを確認してください。現在は3に設定されています)
CPUとRAMの使用状況を確認する
サービスタブ → etrcars-django-service → ヘルスとメトリックタブ → CPU使用率とメモリ使用率（メトリックが過負荷になっていないことを確認してください）
実行中のECSタスクを確認する
サービスタブ → etrcars-django-service → タスクタブ	→ タスクが実行状態であることを確認します (現在 3 つの Django タスクがあります)
実行中のECSタスクログを確認する
サービスタブ → etrcars-django-service → ログタブ → ECSタスクログにエラーがないことを確認します
VPC セキュリティ グループの接続を確認する
EC2 サービス → セキュリティ グループ → etrcars-ecs-sg を選択 → 送信ルール (ポートが RDS に接続できるように適切に設定されていることを確認してください)
EC2 サービス → セキュリティ グループ → etrcars-ecs-sg を選択 → 受信ルール (ポートが ALB に接続できるように適切に設定されていることを確認してください)
EC2 サービス → セキュリティ グループ → etrcars-alb-sg を選択 → 送信ルール (ポートが ECS に接続できるように適切に設定されていることを確認してください)

#### AWS ALB


##### AWS ALB モニタリング

アクセス ログを有効にし、分析のためにログをAmazon S3に保存します。
監視すべき主要な指標:
リクエスト数（ターゲットグループごと）
HTTP 応答コード (4XX、5XX エラー率)
目標応答時間（レイテンシ）
健康なターゲットと不健康なターゲット
接続エラー / 接続拒否
初期アクセスとステータスチェック
IAM ユーザーで AWS マネジメントコンソールにログインします。
EC2 → ロードバランサー → `etrcars-ecs-alb` に移動します。
ALBのステータスを確認する
詳細タブ→状態（アクティブになっていることを確認してください）
[監視] タブで → メトリック (リクエスト数、HTTP エラーなど) を確認します。
ヘルスチェック検証
ターゲットグループのステータスを確認する
EC2 → ターゲットグループ → `etrcars-ecs-target-group` に移動します。
ターゲットタブ→登録したターゲットのヘルスステータスを確認する
ターゲットが正常であることを確認します（現在 3 つのターゲットに接続されています）

##### AWS ALB リスナーとルールの設定チェック

リスナーの設定を確認する
リスナーとルールタブで、次のことを確認します。
プロトコル: `HTTPS`
ポート: 443
デフォルトのアクション: `etrcars-ecs-target-group` に転送
プロトコル: `HTTP`
ポート: `80`
デフォルトのアクション: `etrcars-ecs-target-group` に転送

##### AWS ALB セキュリティ設定チェック

セキュリティグループの設定を確認する
セキュリティタブ→セキュリティグループIDをクリック
セキュリティグループ名が `etrcars-alb-sg` であることを確認してください

## バックアップ管理手順


### 目的

すべての Amazon サービスに、災害復旧、データ保持、ビジネス継続性の要件をサポートする、信頼性が高く、一貫性があり、安全なバックアップが確保されるようにします。

### 範囲

この手順は、組織内の本番環境、ステージング環境、開発環境で使用されるすべての Amazon サービスに適用されます。

### バックアップの種類

自動バックアップ– 定義された保持期間を持つ AWS による毎日のスナップショット。
手動スナップショット– 大きな変更、メンテナンス、またはデプロイメントの前に作成されるアドホック スナップショット。
S3 バックアップ- 定義された保持期間を持つ AWS による東京リージョンから大阪へのリアルタイムレプリケーション。
スクリーンショットによるクラウドフレア設定のバックアップを作成。設定変更を実施したさい、バックアップとして設定画面を記録し、CODIUMフォルダに共有する。
AWSのスクリプト（設定）はGitlabに保存。設定更新時には最新に更新し保存する。

### 手順


#### 自動バックアップ

自動バックアップを有効にし、大阪リージョンにレプリケートします。
自動スナップショットの組織ポリシーに従って保持期間を構成します (毎日 7 回の保持)。
自動更新とパッチ適用のためにメンテナンス ウィンドウが有効になっています。
マイナーアップデートはメンテナンスウィンドウで自動的に実行されますが、メジャーアップデートは手動で行う必要があります。

#### 手動スナップショット

手動でスナップショットを作成してください。
パッチまたはアップグレードの適用
主要な構成の変更
適切な命名規則に従ってスナップショットを保存する

#### AWS RDS バックアップ検証

東京リージョンがダウンした場合の大阪リージョンでのバックアップ復旧手順
大阪エリアに複製されたバックアップデータベース（RDS）を使用して、東京エリアと同じ内容をリストアし、所要時間を確認します。
前提条件
AWS アカウントにログインできる必要があります。
大阪リージョンの最新バックアップ（RDS）を利用する
VPC 構成の設定は gitlab から入力する必要があります。
東京リージョンから大阪リージョンへの RDS データベースのレプリケーションを有効にします。
東京リージョンのバケットから大阪リージョンのバケットへの S3 バケットのレプリケーション ルールを作成します。
別のリージョンで VPC インフラストラクチャ全体を起動するための Terraform スクリプトを作成します。
手順
バックアップデータベースを確認する
AWS マネジメントコンソールにログインします。
必要な複製されたバックアップが大阪リージョンに存在することを確認します。
VPC設定を確認する
以前のインフラストラクチャ設定を作成するためのTerraformスクリプトがあるはずです。Gitlabから最新の設定を確認してください。
大阪リージョンのインフラ再構築のためのスクリプトでVPC設定を変更する
リージョン名や Azs 設定など、大阪リージョンの設定に合わせて、Terraform 構成ファイルの設定を変更します。
大阪リージョンに ECR を作成し、スクリプトを変更して URL を新しい ECR エンドポイントにポイントし、Django アプリが新しく作成された ECR からイメージをプルできるようにします。
ロググループ名などのロググループ設定を大阪リージョンに変更します
EC2 AMI名を大阪リージョンの希望のAMI名に変更します
SSM 名が同じにならないため、SSM ロール名を変更します
S3バケットを大阪リージョンのバケットに変更する
新しい設定を確認して、他の設定に影響がないことを確認します
実行
リソースを復元する
最新の Terraform 設定を適用すると、大阪リージョンに新しいインフラストラクチャが作成されます。(このプロセスは完了するまでに数分かかる場合があります。)
RDSインスタンスを復元する
RDS コンソールに移動します。
「自動バックアップ」を選択し、「レプリケート」タブを選択して、東京リージョンからレプリケートされたデータベースを選択します。
システムスナップショットを選択し、大阪リージョンの新しい VPC 構成を選択してデータベースを復元します。(このプロセスは完了するまでに数分かかる場合があります。)
CloudflareのDNS設定を変更する
Cloudflareアカウントにログイン
ドメインを選択し、DNS設定に進みます
CNAME エンドポイント URL を、大阪リージョンで新しく作成した ALB エンドポイント URL に変更します。
AWS ACM 内でドメインの証明書を作成し、ALB リスナーの certificate_arn で ARN を使用します。
新しいRDSエンドポイントへの変更
RDS の復元が完了したら、terraform スクリプトの Django ECS で新しい RDS エンドポイントに変更します。
新しい Terraform 設定を再度適用します。
Gitlab Pipelineの環境変数を変更する
パイプラインを再度実行する際に、Gitlab 環境変数の ECR レジストリとリージョン変数を変更します。
シークレットマネージャーの値を追加する
大阪リージョン内にシークレット マネージャーの値を追加し、値を取得するためのシークレット マネージャーのポリシーをアタッチします。
接続設定を確認してください
復元された ECS とデータベースが正しいサブネットまたはセキュリティ グループに配置されていることを確認します。
Gitlab パイプラインが正常に実行され、データベースが移行されていることを確認します。
テストと検証
データや画像など、復元されたリソースが正しく機能するかどうかをテストします。
アプリケーションの接続性と機能性を確認します。
注意事項
VPC 設定の手動検証は必須です。事前に設定内容を文書化することを強くお勧めします。
復元プロセス中に他のリソースが影響を受けないことを確認します。

#### AWS ECS バックアップ検証

回復方法
上記のシナリオを確認した後に何か問題がある場合は、ECS を回復するために試すことができる方法があります。
Elastic Container Service → etrcars-ecs-cluster → etrcars に移動します。
最後のパイプラインを再実行する
Gitlabに移動→最後のデプロイメントパイプラインを再実行
新しい展開を強制する
サービスタブ → etrcars-django-service → サービスの更新 → 新しいデプロイメントを強制
希望タスク数の変更（システムが過負荷になった場合）
サービスタブ → etrcars-django-service → サービスの更新 → 希望カウント → 希望するカウント値を設定
スケーリングポリシーを変更する（システムが過負荷になった場合）
サービスタブ → etrcars-django-service → サービスの更新 → サービスの自動スケーリング → 最小および最大タスクとスケーリングポリシーの設定を変更します
回復後
ECS を回復した後、次の点を検証する必要があります。
システムが正常に動作していることを確認する
ウェブサイトをチェックして、データが正しく表示されているか確認してください
iPadの通常の操作機能が正常に動作しているかどうかを確認します

#### AWS EC2 バックアップ検証

回復方法
上記のシナリオを確認した後に何か問題がある場合は、EC2 を回復するために試すことができる方法があります。
新しいEC2インスタンスを作成する
EC2 → インスタンス → インスタンスの起動に移動します
（名前とタグ）名前: etrcars_bastion_new

#### 災害復旧のための緊急時対応およびエスカレーション計画

上記のいずれかの手順を実行した後で復元に失敗した場合は、緊急時対応計画として次の手順に従います。
障害検出
- 手順が正常に完了しない場合は、エラーの詳細 (ログ、スクリーンショットなど) を記録します
- 問題が一時的なもの (再試行可能) か構造的なもの (続行不可) かを確認します。
即時のアクション
- 初めて失敗したら手順を再試行します
- 再試行も失敗した場合は、データの破損やダウンタイムの延長を回避するために現在の手順を停止します。
エスカレーションとレポート
- CODIUM プロジェクト マネージャーに通知
- 30 分以内または 2 回の再試行で回復できない場合、CODIUM プロジェクト マネージャーは エスブランド および CODIUM 管理にエスカレーションします。 
- 実行されたすべてのアクションをインシデント Trello カードに記録します。
代替/緊急対応フロー
- 以前の安定したバックアップスナップショットのバックアップを試す
- 別のリージョンで復元を試行する
- 可能な場合は回避策を探す
- 他のシニアエンジニアと協力して問題を解決する
- 有効な回避策またはロールバックが不可能な場合、復旧の試行は中止されます。今後の決定は、CODIUMまたはエスブランドの経営陣が行います。
障害後レビュー
- エンジニアリングチームが根本原因分析を実施
- すべての障害の詳細が文書化されます
- CODIUMはタイムライン、詳細、予防手順を含む事後分析ドキュメントを公開します

## リリースおよび変更管理手順


### GitLabにおけるコード管理とマージ手順


#### 目的

コードの変更を管理、レビュー、マージする方法を標準化し、コードの品質を維持し、安定したリリースを確保します。

#### 手順

分岐戦略
2つの環境（開発と本番）に基づいて2つの主要なブランチがあります
変更リクエストまたは機能に基づいて新しいブランチが作成されます。
機能または修正が準備できたら、マージされたリクエストが作成され、開発ブランチにマージされます。
その後、本番環境へのデプロイが完了すると本番環境ブランチにマージされます。
マージリクエストとコードレビュー（MR）
開発者はコードを GitLab にプッシュし、マージ リクエストを作成します。
マージリクエストはシニア開発者によってレビューされ、開発サーバーの展開のために承認されます。
シニア開発者が利用できない場合は、別のシニア開発者またはプロジェクト マネージャーがレビューします。
マージ
シニア開発者またはマネージャーである承認されたメンテナーのみが MR をマージできます。
マージ後、Gitlabはコードパイプラインを実行し、特定の環境にコードをデプロイします。
タグ付けとリリース
本番環境への展開では、シニア開発者またはマネージャーが必要なコードを本番環境ブランチにマージします。
その後、ブランチはタグ付けによってバージョン管理されます。タグ付けにより、パイプラインが本番環境へのデプロイを開始します。
シニア開発者またはマネージャーである承認されたメンテナーのみがバージョンにタグを付けることができます。
シニア開発者が利用できない場合は、別のシニア開発者またはプロジェクト マネージャーがレビューします。
ロールバック
重大な障害が発生した場合、以前の安定した Git タグ バージョンが再デプロイされます。

### AWS ECSでのリリース確認


#### 目的

AWS ECS にデプロイされた新しいリリースが正常に適用され、正しく機能し、サービスの中断が発生しないことを確認します。

#### 手順


##### ECS サービスのステータスを確認する

ECS → クラスター → etrcars-ecs-cluster → サービスに移動します。
サービスのステータスがアクティブであり、最後のデプロイが完了であることを確認します。

##### 実行中のタスクを確認する

ECS → クラスター → etrcars-ecs-cluster → etrcars-django-service → タスクに移動します。
タスクの数が希望の数と同じであることを確認します。
タスクのステータスが実行中であることを確認します。
タスク定義が最新のタスク定義であることを確認します。

##### システムを確認する

アプリケーションが正しく実行されており、必要な機能が展開されていることを確認します。

## 障害対応手順


### 目的

実行時例外により本番 Web アプリケーションの実行が停止した場合に、標準化された対応計画を提供し、迅速な回復、最小限のダウンタイム、根本原因の分析を保証します。

### 範囲

実行時エラーが発生し、サービス停止やパフォーマンス低下を引き起こす、運用環境 (クラウド ホスト) に展開されているすべての Web アプリケーションに適用されます。

### 手順


#### 検出と初期トリアージ

クラウド監視システムからアラートを受信する
問題を確認する
インシデントの重大度を分類する
CloudWatchログからログを確認する

#### 封じ込めとサービスの復旧

バックアップ管理手順の手順に従ってAWSサービスを復元します。

#### AWS RDSフェイルオーバー

RDSフェイルオーバーの場合、RDSは自動的に次の処理を実行します。
スタンバイ レプリカが昇格され、新しいプライマリ インスタンスになります。
別のアベイラビリティーゾーンに新しい同期スタンバイが自動的に作成されます。
フェイルオーバーは通常60 ～ 120 秒以内に完了しますが、データベースのアクティビティと回復状況に応じて時間は異なる場合があります。
フェイルオーバー時の操作手順
フェイルオーバーが発生すると、CloudWatch および RDS イベントでアラートが発生します。
アプリケーションは同じ RDS エンドポイントを使用して自動的に再接続します。
手動介入は不要

#### 根本原因分析（RCA）

Cloudwatch からログを収集します。
どのサービスにエラーの原因があるかを特定します。
障害時間と最近の変更を相関させる:
新しい展開
構成の更新
外部サービスへの依存関係
開発チームを関与させて障害を再現し、分析します。

#### 事後レビュー

48 時間以内に非難の余地のない事後検証を実施します。
書類：
イベントのタイムライン（検出→解決）
根本原因と要因
回復に向けて取られた措置
予防策（例：コード修正、監視アラートの改善、フェイルオーバーの強化）
エスブランド、ENEOSと概要を共有。
