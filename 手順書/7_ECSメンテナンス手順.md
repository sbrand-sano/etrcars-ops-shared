ECSメンテナンス手順
- IAM ユーザーでAWS マネジメントコンソールにログインします。
- Elastic Container Service → etrcars-ecs-cluster → etrcarsに移動します。
- ECS サービスのステータスを確認します
- [サービス]タブ-> [ステータス] ( [アクティブ]と表示されていることを確認します)
- ECS タスク数
を確認します- [サービス]タブ-> [etrcars - django -service] -> [構成とネットワーク]タブ -> [Auto Scaling] -> [Desire Count] (0 より大きいことを確認してください。現在は 3 に設定されています)
- CPU と RAM の使用状況を確認します
- [サービス]タブ-> [etrcars - django -service] -> [ヘルスとメトリック]タブ -> [CPU 使用率] と [メモリ使用率] (メトリックが過負荷になっていないことを確認します)
- 実行中の ECS タスクを確認します
- [サービス]タブ-> [etrcars - django -service] -> [タスク]	タブ -> タスクが実行状態であることを確認します (現在 3 つの Django タスクがあります)
- のECS タスク ログを確認します
- [サービス]タブ -> [etrcars - django -service] -> [ログ]タブ -> ECS タスク ログにエラーがないことを確認します
- 接続用のVPCセキュリティグループを確認
- EC2サービス->セキュリティ グループ-> etrcars - ecs -sg を選択 -> 送信ルール（ポートが RDS に接続できるように適切に設定されていることを確認してください） 
- EC2 サービス->セキュリティ グループ-> etrcars - ecs -sg を選択 -> 受信ルール（ポートが ALB に接続できるように適切に設定されていることを確認してください） 
- EC2 サービス->セキュリティ グループ-> etrcars - alb -sg を選択 -> 送信ルール (ポートが ECS に接続できるように適切に構成されていることを確認してください)
回復方法
上記のシナリオを確認した後に何か問題が発生した場合、ECS を回復するために試すことができる方法は次のとおりです。
- Elastic Container Service → etrcars-ecs-cluster → etrcarsに移動します。
- 最後のパイプラインを再実行
- Gitlab に移動 -> 最後のデプロイメント パイプラインを再実行
- 新しいデプロイメントを強制
-サービスタブの下-> etrcars - django -service -> サービスの更新 -> 新しいデプロイメントを強制
- 希望タスク数の変更 (システムが過負荷になっている場合) 
-サービスタブ-> etrcars - django -service -> Update Service -> Desire Count -> 希望するカウント値を設定します
- スケーリング ポリシーを変更する(システムが過負荷になっている場合) 
- [サービス]タブ-> etrcars - django -service -> [サービスの更新] -> [サービスの自動スケーリング] -> 最小および最大タスクとスケーリング ポリシーの設定を変更します
リカバリ後ECS
をリカバリした後、以下の点を確認する
必要があります。
- システムが正常に動作していることを確認します
- ウェブサイトをチェックして、データが正しく表示されているかどうかを確認します - iPad をチェックして、通常の操作機能が正常に動作しているかどうかを確認します
