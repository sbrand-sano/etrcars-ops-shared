1　定期メンテナンス手順
1.1 目的
Security Hub の統合ダッシュボードと自動パッチ管理統合により、すべての AWS サービスにわたるセキュリティの検出結果、脆弱性、脅威を一元的に管理し、修復します。
1.2 範囲
すべての AWS アカウントにわたる統合サービス (GuardDuty、AWS Inspector、AWS Config、IAM Access Analyzer など) からのすべての Security Hub の検出結果をカバーします。
1.3 手順
1.3.1 AWS 環境
承認された AWS メンテナンス ウィンドウ内で EC2、ECS、および RDS インスタンスにセキュリティ パッチを適用します。
EC2 の自動化サービス (SSM Patch Manager) を使用します。
機能スモーク テストを使用して、パッチ適用後のインスタンス/サービスの健性を検証します
1.3.2 GitLab／MDM／ファイアウォール
重要なパッチについては、ベンダーのアドバイザリと GitLab リリース ノートを確認してください。
パッチを適用する前に、構成とリポジトリをバックアップします。
関係者にダウンタイムを通知し、アップグレード後のシステム動作を検証します。
Meraki MDM は会社の PC にパッチを適用するために使用されます。
会社のネットワークで使用されているファイアウォールのファームウェア更新を確認し、必要に応じてパッチを適用します。
1.3.3 操作確認
iPadアプリケーションやWeb管理画面からアクセシビリティや機能を確認します。
確認結果をTrelloの公式メンテナンスログに記録します。
2　予防保守手順
2.1 目的
潜在的なリスクを早期に特定し、システムの信頼性と容量を積極的に維持します。
2.2 範囲
AWS および CODIUM 内部環境全体の監視、パフォーマンステスト、パッチステータスの検証に適用されます。
2.3 手順
ログ分析
Cloudflare、AWS GuardDuty、Cloudwatch、AWS Inspector などのさまざまなツールで検出された異常を定期的に分析します。
潜在的なセキュリティまたはパフォーマンスのリスクが検出された場合は、調査結果をエスカレーションします。
セキュリティパッチステータスのレビュー
AWS Config、Security Hub、MDM コンプライアンスダッシュボードを確認し、必要に応じてパッチを適用します
3　障害エスカレーション手順
3.1 目的
大規模なシステム障害や停止が発生した場合のエスカレーションおよび回復プロセスを定義します。

SBrand および ENEOS トレーディング本社向けに CODIUM がサポートする AWS ワークロード、社内システム、アプリケーションをカバーします。

根本原因分析
講じられた措置（一時的および永続的）
再発防止のための推奨事項
合意された SLA 期間内に SBrand と ENEOS にレポートを提出します

4.1 影響調査および停止要否の判定
パッチ適用前に以下 5 項目を確認し、作業方式（無停止／要停止）を決定する。

#### 4.1.1 対象リソースの種類

EC2：OS／カーネル更新が発生する場合は停止可能性あり
ECS：新しい task 定義による再デプロイが必要
RDS：minor update により数秒～数十秒の再起動が発生

#### 4.1.2 再起動の必要性（判断基準）

以下のいずれかに該当した場合は「再起動あり」と判定する。
Patch Manager 結果に RebootRequired=True
パッチ名に「kernel」「openssl」等の再起動必須コンポーネントを含む
ベンダー説明文に restart required の記載
SecurityUpdate（OS系）がプロセスに依存する場合
→ 再起動あり＝停止計画が必須

#### 4.1.3 デプロイを要するワークロード（ECS）

Inspector にて CVSS ≥ 7.0 → コンテナ再ビルド
（9.0以上は即適用。申請を行い対応する）
新バージョン task 定義作成 → Rolling Update
Blue/Green デプロイのため停止なし

#### システム停止の可能性（判断基準）


| リソース | 停止扱い基準 | 停止要否 |
|---|---|---|
| EC2 | 再起動必須パッチを含む | 停止要 |
| ECS | Rolling Update 可能 | 停止不要 |
| RDS | minor update で再起動 | 停止扱い（22時以降） |


#### 4.1.5 特約店利用時間との調整

特約店ピーク：9:00〜18:00
作業許可帯：22:00〜翌5:00
ENEOS トレーディング承認が優先
対象リソースの種類（EC2/ECS/RDS）
再起動の必要性
デプロイが必要なワークロードの有無
システム停止が発生する可能性の有無
特約店の利用時間帯（ピーク：9～18時）との衝突

5.3 セキュリティパッチ（RDS）

### 5.3.1 目的

すべての Amazon RDS インスタンス（本番・開発）が最新のセキュリティパッチおよびエンジンバージョンで維持され、脆弱性の発生を防止する。

### 5.3.2 範囲

本番環境 RDS（PostgreSQL）
開発環境 RDS

### 5.3.3 手順


### ① パッチ情報の収集（具体化）

AWS コンソール → RDS → 「インスタンス」を開く
影響範囲を確認
‐「保留中のメンテナンス」（Pending maintenance）
‐「エンジンバージョン」
RDS リリースノート、AWS Security Bulletin を確認
パッチ種別を分類
- Security Patch（優先適用）
- Minor Version Upgrade（再起動あり）
- Parameter Group Update（再起動の可能性あり）

### ② 影響評価（判断基準追加）

以下を確認し、停止要否を明確に判断する：

#### 2-1. 再起動の有無（必須確認項目）

Minor Update → 再起動あり（数秒〜数十秒の接続断）
Parameter Group の変更 → 再起動の可能性
Engine Patch → 再起動あり

#### 2-2. 互換性影響（具体的項目）

下記項目を RDS リリースノートで確認：

| 項目 | 内容 |
|---|---|
| SQL文の互換性 | 廃止予定（Deprecation）コマンドの確認 |
| 拡張機能の対応可否 | pgcrypto、uuid-osspなど利用中のExtが対応しているか |
| 認証方式 | SCRAM/MD5の互換性変更 |
| パフォーマンス | インデックス再構築が必要になるケースがあるか |


#### 2-3. システムへの影響度分類

High　CVSS 7.0以上を　30日以内に適用する（9.0以上は即適用。申請を行い対応する）
③ スケジューリング（明確化）
作業実施は 22:00〜翌5:00
特約店稼働時間（9〜18時）は作業禁止
メンテナンスウィンドウ（AWS設定）内に合わせる
必ず ENEOS トレーディング承認を取得
④ バックアップ（具体化）
4-1. Snapshot 作成手順
RDS コンソール
対象インスタンス →「スナップショット」→「スナップショット作成」
完了を「Available」で確認
4-2. 自動バックアップ保持の確認
Retention（7～30日）
PITR（Point-In-Time Recovery）有効か確認
⑤ パッチ適用（具体化）
手順については、『RDS パッチ適応詳細 _20260108』を参照し実施

5.6 セキュリティパッチ（MDM／エンドポイント）

#### 目的

すべての従業員のエンドポイント (ラップトップ、デスクトップ、モバイル デバイス) にパッチが適用され、会社の MDM (モバイル デバイス管理) ポリシーに準拠していることを確認します。

#### 範囲

すべての企業エンドポイントは CODIUM の MDM プラットフォームに登録されています。

#### 手順

エンドポイントのパッチ要件を MDM プラットフォームと定期的に同期します。
MDM を通じてセキュリティ パッチ (macOS パッチ、アプリケーション アップデート) を自動的に展開します。
- MDMコンソールへアクセスする
- MDMコンソール内、ソフトウエアの更新設定が自動になっていること確認
- ソフトウエアを確認し、アプリケーションの最新バージョンをAPPストアで確認
- MDM上、実機上のバージョンが合致しているか確認。
- アップデートをしていないものがある場合には、その理由を確認しアップデートの検討する
6　Trello を使用したリリース提出・承認手順
進行中のタスクと履歴はすべて Trello に保存されます。

#### 手順

提出
バグや変更があった場合に備えて、新しい Trello カードが開かれます。
エスブランド は、必要な説明を与えて、CODIUM に修正または変更を割り当てます。
修正
CODIUM はカード内の説明を確認し修正します。
開発サーバーでタスクを修正した後、CODIUMはスクリーンショットまたはビデオを添えてそれぞれのTrelloカードにコメントします。
修正内容、原因、修正方法が記載されます。
レビュー
この修正はエスブランドによって検討されます。
承認
審査が完了すると、カードはスケジュールに従って本番環境に展開することが承認されます。
展開済み
導入が成功した後、エスブランドは本番環境での機能確認を行います。
その後カードは閉じられます。
